FrontierAI — Architecture Notes

bretmckinney.com · Bret McKinney · June 2026

FrontierAI is a 21-agent autonomous research system running 24/7 on a local machine. It has completed over 8,500 research objectives without human intervention. This document covers the architectural decisions that made that possible — why each layer is designed the way it is, what alternatives were considered, and what the trade-offs look like in practice.

This is not a tutorial. It is a record of decisions made while building a production system under real constraints: local compute, limited budget, no DevOps team, and a requirement that the system run continuously without manual intervention.

The Problem

Macro investment research is continuous. Markets move on Fed communications, Treasury flows, geopolitical events, credit spreads, and dozens of other signals simultaneously. Staying current requires monitoring all of them, synthesizing relationships between them, and updating a structured knowledge base as conditions change.

A human can do this for a few hours a day. A single AI model can process a prompt but cannot maintain state, schedule itself, or pursue a research agenda autonomously. Neither solution produces continuous, compounding intelligence.

The requirement: a system that runs without human prompting, generates its own research objectives, executes them, stores the findings in a way that builds on prior work, and delivers validated output to a subscriber-facing platform. FrontierAI is that system.

High-Level Architecture

The system has four layers:

  • Objective queue — what needs to be researched
  • Agent layer — specialized agents that execute research, store findings, and synthesize output
  • Memory layer — persistent knowledge base that agents read from and write to
  • Publishing pipeline — validated output delivered to Avital Research

The orchestrator sits above all four layers. It runs a master loop on a 600-second cadence, pulls objectives from the queue, dispatches them to the appropriate agent, handles failures, and manages the weekly rhythm of dedicated regime classification work.

Key design principle: No agent has global state. Each agent receives its task, executes it, writes to the memory layer, and exits. The orchestrator maintains continuity, not the agents. This makes individual agents replaceable without disrupting the system.

Agent Design

Specialization Over Generality

FrontierAI uses 21 specialized agents rather than one general-purpose agent for the same reason a newsroom uses reporters, editors, and fact-checkers rather than one person doing everything. Each role requires a different cognitive posture.

The agent roster by function:

  • Research, Reasoning, Synthesis, Scout, Curator — the research cluster. These agents gather raw information, assess credibility, and produce structured findings.
  • Memory, Knowledge, Report Gen, Valuation, News Scanner — the knowledge cluster. These agents store, index, retrieve, and format findings for downstream consumption.
  • Publishing, Social, Avital Sync, Integration Bridge, Notifications — the communications cluster. These agents move validated output to external surfaces.
  • Orchestrator, Chief of Staff, Health Monitor, Catalyst Monitor, Rhythm Engine, Execution — the operations cluster. These agents manage scheduling, system health, and the trading paper mode.

Specialization has a cost: coordination complexity. The orchestrator must know which agent handles which task and route accordingly. This routing logic is centralized in the orchestrator rather than distributed across agents, which makes the system easier to reason about when something breaks.

Model Selection

Not all agents use the same model. This is an intentional cost and performance optimization.

  • Gemini 2.5 Flash handles research agents — high volume, fast turnaround, web search integration via Tavily. Cost matters here because research agents run constantly.
  • Claude handles execution and synthesis tasks that require deeper reasoning, nuanced judgment, or decisions about whether to deepen or pivot on a research thread. More expensive, used selectively.

Using a single expensive model for every task would make the system economically unviable at the volume FrontierAI operates. The right tool for each job is a cost discipline, not a compromise.

Trade-off acknowledged: Gemini 2.5 Flash and Claude do not produce identical output quality on analytical tasks. Synthesis tasks that were previously assigned to Flash occasionally produced shallower analysis. The current split — Flash for research volume, Claude for synthesis depth — reflects where quality differences are consequential versus acceptable.

Self-Correction

Research agents can get stuck. A research objective may return low-quality web results, hit a dead end, or produce findings that contradict the existing knowledge base. The system handles this through the ExecutionAgent.

When a research agent flags uncertainty or low confidence, the ExecutionAgent receives the objective with context about what was tried. It decides: deepen the current approach, pivot to a different angle, or mark the objective as requiring human review. This prevents the system from silently producing low-quality output when the research environment is noisy.

8,500+ completed objectives means the self-correction loop has run thousands of times. It is not a theoretical feature — it is what keeps the system running through bad data days, API outages, and ambiguous research questions.

Memory Architecture

Two-Layer Storage

FrontierAI uses two storage systems with distinct purposes:

  • PostgreSQL for structured data — research objective records, agent run logs, regime classifications, pillar scores, publication records. Queryable, relational, durable.
  • Pinecone for vector storage — embeddings of research findings, synthesis outputs, and regime narratives. Enables semantic retrieval across the full knowledge base.

The split is intentional. PostgreSQL answers structured questions: what objectives ran last Monday, what was the confidence score on the Goldilocks classification in March, which agents failed in the last 24 hours. Pinecone answers semantic questions: what prior research is most relevant to a new objective about credit spread compression.

Why Pinecone Over Self-Hosted

Three options existed: Pinecone managed service, pgvector extension on the existing PostgreSQL instance, or a self-hosted Weaviate or Qdrant instance.

  • pgvector was ruled out early. FrontierAI runs on local hardware with limited RAM. Running high-dimensional vector operations on the same Postgres instance handling structured data would create resource contention during peak research runs.
  • Self-hosted vector database was ruled out for the same reason — local hardware constraint. Also adds operational overhead that a single-operator system cannot absorb.
  • Pinecone managed service: higher per-query cost than self-hosted, but zero operational overhead, no hardware contention, and purpose-built performance for the retrieval pattern FrontierAI uses.

At 8,500+ research objectives, the Pinecone index contains tens of thousands of embeddings. Retrieval latency is consistent. The cost trade-off has been worth it.

Reusable pattern: The two-layer memory architecture — relational for structured queries, vector for semantic retrieval — is a generalizable pattern for any AI system that needs both operational observability and knowledge retrieval. The split eliminates the temptation to use a vector database as a general-purpose store, which degrades both retrieval quality and operational clarity.

Embedding Strategy

Research findings are embedded at the paragraph level, not the document level. Document-level embeddings produce vectors that average meaning across the whole document, which dilutes specificity. Paragraph-level embeddings preserve distinct claims, which is what retrieval needs to surface when a synthesis agent is looking for evidence on a specific point.

The trade-off: more embeddings means higher storage cost and more retrieval candidates to re-rank. At the current scale this is manageable. At 10x scale it would require a chunking strategy review.

Orchestration

Queue-Based vs. Event-Driven

FrontierAI uses a queue-based orchestration pattern via Upstash Redis rather than an event-driven pattern. This was a deliberate choice with a specific trade-off.

Event-driven systems are more reactive — an agent publishes an event and downstream agents respond immediately. Lower latency, higher coupling. Queue-based systems are more decoupled — the orchestrator controls dispatch timing, can throttle under load, and has a single source of truth about what is in progress.

For a system running 24/7 on local hardware with external API rate limits, decoupling is more valuable than latency. The 600-second master loop cadence is not a limitation — it is a feature. It prevents the system from hammering APIs during rate limit windows and gives the orchestrator time to assess system health before the next dispatch cycle.

Failure Handling

Three failure modes matter in practice:

  • Agent crash: the orchestrator detects absence of a completion signal after a timeout threshold and requeues the objective. Health Monitor surfaces repeated failures on specific objective types.
  • API failure: external API outages (Tavily, Anthropic, Google, Pinecone) are caught at the agent level. Agents return a structured failure result rather than throwing. The orchestrator decides whether to retry immediately, defer, or escalate to human review.
  • Data quality failure: synthesis output that fails confidence thresholds is flagged by the Catalyst Monitor rather than published. The SAIR scorecard requires manual review before going live.

The human-in-the-loop gate is not a limitation of the architecture. It is a design choice. Fully autonomous publishing to a subscriber-facing financial intelligence platform would be irresponsible. The system is built to maximize autonomous research while keeping a human accountable for what gets published.

Weekly Rhythm

FrontierAI operates on a structured weekly schedule:

  • Monday: 8 dedicated regime classification objectives. The regime engine re-evaluates all 7 pillars with fresh FRED data.
  • Wednesday: 8 SAIR (Sovereign AI Industrial Revolution) framework objectives.
  • Continuous: general research objectives across macro, crypto, geopolitics, and credit markets.

The rhythm exists because some research has temporal structure. Regime classification is most valuable when it is consistent — same methodology, same cadence, comparable to prior periods.

Integration Architecture

FrontierAI to Avital Research

FrontierAI and Avital Research are separate systems that communicate through a defined integration point. The AvitalSyncAgent runs after regime classification objectives complete. It reads synthesis output from the memory layer, formats it for the Avital API, and pushes updated regime classifications, pillar scores, and portfolio guidance to the Avital backend.

The integration is one-directional: FrontierAI writes to Avital, Avital does not write back. This keeps the systems decoupled — Avital can be rebuilt or migrated without touching FrontierAI's core research pipeline.

External Data Sources

  • FRED — macroeconomic indicators. Primary source for regime pillar scoring.
  • Tavily — web search API for current events and non-FRED data.
  • Polygon.io — market data for stress indicators and asset price feeds.
  • CoinGecko — crypto market data for digital asset coverage.

Each integration has a failure mode handler. FRED outages are rare but consequential — a Monday regime run that cannot access FRED data is deferred rather than run on stale data.

Security Architecture

FrontierAI runs on local hardware. The ClawFactory security substrate that packages FrontierAI for external distribution applies the same principles to that deployment context:

  • WSL2 sandbox isolates the agent runtime in a scoped Linux user environment.
  • nftables egress firewall restricts outbound network access to the specific Linux UID running the agents.
  • DPAPI stores API keys in Windows Credential Manager — encrypted at rest, bound to the Windows user account.
  • Loopback-only gateway binding prevents the agent gateway from being accessible outside localhost.
  • Kill Switch performs instant teardown — terminates processes, clears the gateway, revokes the active session.

The security architecture was designed before ClawFactory existed as a product. The requirement was personal: run AI agents locally without creating an exfiltration risk or a persistent attack surface. The productization came after the security model was proven in production.

What Would Be Different at Enterprise Scale

FrontierAI is a single-operator system built for continuous research on local hardware. Several architectural decisions would change at enterprise scale:

  • Orchestration would move from a local Python process to a managed workflow engine — Temporal, Prefect, or Airflow — with distributed execution, built-in retry semantics, and observability tooling.
  • The queue would move from Upstash Redis to a more durable message broker — SQS, Kafka, or Pub/Sub — depending on throughput and ordering requirements.
  • The two-layer memory architecture would scale horizontally. PostgreSQL would move to RDS or Cloud SQL. Pinecone would remain managed but would require a namespace strategy for multi-tenant isolation.
  • Model selection would require a formal cost-quality framework rather than operator intuition. At enterprise volume, model cost per token is a budget line item, not an afterthought.
  • The human approval gate would become a structured review workflow rather than a manual check. The gate stays — the tooling around it changes.

The core architectural decisions — specialization over generality, two-layer memory, queue-based orchestration, decoupled integration, human gate on publish — are the same decisions that would survive the transition to enterprise scale. The implementation details change. The design principles do not.
